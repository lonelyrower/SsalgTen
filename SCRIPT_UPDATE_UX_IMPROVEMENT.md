# 脚本更新用户体验改进

## 📋 问题描述

**原有行为**：
```bash
请选择操作 [0-14]: 3
🔄 脚本自更新
确认更新脚本到最新版本? [Y/N] (默认: Y): y
[INFO] 下载最新版本...
[SUCCESS] ✅ 脚本已更新到最新版本 (unknown)
[INFO] 重新运行以使用新版本
[SUCCESS] 操作完成，程序退出
```

**问题**：
- ❌ 脚本更新后直接退出程序
- ❌ 用户需要手动重新运行脚本
- ❌ 用户体验中断，需要额外步骤
- ❌ 与其他操作（如部署、更新系统）的退出行为一致，但不合理

**用户期望**：
- ✅ 脚本更新后自动重新加载
- ✅ 继续显示菜单界面
- ✅ 无需手动重新运行
- ✅ 流畅的用户体验

---

## ✨ 改进方案

### 1. **自动重新加载脚本**

使用 `exec` 命令重新执行更新后的脚本：

```bash
case "$choice" in
    3) 
        # 脚本更新：更新后重新执行脚本
        self_update
        if [[ $? -eq 0 ]]; then
            log_info "重新加载更新后的脚本..."
            sleep 1
            exec "${BASH_SOURCE[0]}" "$@"  # 重新执行脚本
        fi
        ;;
```

**关键技术点**：
- `exec "${BASH_SOURCE[0]}"` - 替换当前进程，执行新版本脚本
- `"$@"` - 传递原始命令行参数
- 仅在更新成功时重新加载（`$? -eq 0`）

---

### 2. **区分脚本更新和其他操作**

其他操作仍然保持退出行为：

```bash
if [[ "$choice" != "0" ]] && [[ "$choice" != "3" ]]; then
    echo
    # 操作完成后直接退出，不返回菜单（脚本更新除外）
    log_success "操作完成，程序退出"
    exit 0
fi
```

**逻辑**：
- 选项 0：退出 ✅
- 选项 3：脚本更新 → 自动重新加载 ✅
- 其他选项：完成后退出 ✅

---

### 3. **移除多余提示信息**

更新 `self_update()` 函数，移除"重新运行以使用新版本"提示：

**修改前**：
```bash
else
    log_success "✅ 脚本已更新到最新版本 ($new_version)"
    log_info "重新运行以使用新版本"  # ← 移除
fi
```

**修改后**：
```bash
else
    log_success "✅ 脚本已更新到最新版本 ($new_version)"
    # 不再提示"重新运行"，因为交互式菜单会自动exec重新加载
fi
```

## 🛠️ 技术实现细节

### exec 命令原理

`exec` 是 bash 内建命令，用于替换当前 shell 进程：

```bash
exec "${BASH_SOURCE[0]}" "$@"
```

**工作流程**：
1. 下载并安装新版本脚本到 `${BASH_SOURCE[0]}`
2. 调用 `exec` 替换当前进程
3. 新进程使用更新后的脚本代码
4. 继续执行 `main()` 函数
5. 显示更新后的交互式菜单

**关键点**：
- ✅ 进程ID保持不变
- ✅ 环境变量保留
- ✅ 无需等待旧进程退出
- ✅ 立即使用新代码

---

### 参数传递

```bash
exec "${BASH_SOURCE[0]}" "$@"
```

- `"$@"` - 传递所有原始命令行参数
- 保证重新加载后参数一致性
- 支持 `--dir`, `--force` 等选项

**示例**：
```bash
# 原始命令
ssalgten --dir /opt/ssalgten

# 选择"脚本更新"后重新加载
exec ssalgten --dir /opt/ssalgten  # 参数保留
```

---

### 错误处理

```bash
self_update
if [[ $? -eq 0 ]]; then
    log_info "重新加载更新后的脚本..."
    sleep 1
    exec "${BASH_SOURCE[0]}" "$@"
fi
```

**安全机制**：
- 仅在更新成功时重新加载
- 如果下载失败，不会 exec
- 如果验证失败，保持当前版本
- 用户可以重试或退出
